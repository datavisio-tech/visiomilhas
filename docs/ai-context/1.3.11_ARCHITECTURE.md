VisioMilhas — 1.3.11 — Reavaliação arquitetural do motor de milhas

Objetivo

- Documentar análise do estado atual (schema APP e ações de mutação) e propor um plano técnico incremental para introdução de `mile_point_lots` (lotes) e motor FIFO, sem aplicar migrations nesta etapa.

1. Estado atual (inspeção de `db/app/schema.ts`)

- Tabelas relevantes já presentes:
  - `program_accounts` (saldo atual e campos de custo agregado: `currentPointsBalance`, `currentCostBasisCents`, `current_avg_cost_per_thousand_cents`).
  - `mile_entries` (registro genérico de entradas/saídas, com `points`, `costBasisCents`, `amountCents`, `direction`, `type`).
  - `mile_purchases` (compra de pontos; tem `points`, `amountCents`, `totalCostCents`, `costPerThousandCents`).
  - `mile_sales` (venda de pontos; possui `points`, `revenueCents`, `costBasisCents`, `profitCents`).
  - `mile_transfers` (transferências; possui `pointsSent`, `bonusPoints`, `transferredCostBasisCents`, `destinationCostPerThousandCents`).

- Observação crítica: não existe persistência de lotes (`mile_point_lots`) — logo, cálculos FIFO explícitos não podem ser reproduzidos a partir do schema atual sem histórico externo.

2. Gaps em relação ao blueprint ledger + point_lots

- Falta de tabela de lotes por origem (purchase/airline/bonus) com: quantity_remaining, original_points, cost_basis_cents, issued_at, expires_at, source_type, source_id.
- Ausência de referência de consumo por lote em `mile_entries` (ex.: `consumed_lot_id`, `consumed_points_from_lot`).
- Vendas e transferências atualmente registram `cost_basis_cents` agregado, mas não o custo calculado por lote FIFO; isso impede auditoria e reversão granular.
- Índices necessários para performance em consultas: `account_id`, `organization_id`, `program_id`, `available_points DESC, expires_at ASC` em `mile_point_lots`.

3. Proposta de modelagem (apenas proposta — sem migrations)

- Nova tabela proposta: `mile_point_lots`
  - id (serial PK)
  - organization_id
  - program_id
  - account_id
  - source_type (enum: purchase|bonus|adjustment|transfer_in)
  - source_id (varchar) — referência externa (ex.: purchase id)
  - original_points (integer)
  - points_remaining (integer)
  - cost_basis_cents (integer)
  - cost_per_thousand_cents (integer)
  - issued_at (timestamp)
  - expires_at (timestamp nullable)
  - status (varchar)
  - metadata (json)
  - created_at, updated_at

- Alteração/Extensão sugerida em `mile_entries`:
  - adicionar `consumed_lot_id` (nullable)
  - adicionar `consumed_points` (quando uma entry consome parte de um lote)
  - adicionar `lot_snapshot` (json com histórico do lote ao consumo)

- Alterações conceituais: `mile_sales.costBasisCents` deve ser preenchido a partir do somatório dos lots consumidos (FIFO), persistindo também quais lotes foram usados (para auditoria e reversões).

4. Motor FIFO (comportamento esperado)

- Regras gerais:
  - Compras: geram um `mile_purchase` + `mile_point_lots` (1 ou mais se parcelado) + `mile_entries` creditando pontos.
  - Venda: consumir lotes do `from` account por ordem FIFO (considerar `expires_at` e regras de priorização), gerar `mile_sales` com `costBasisCents = sum(lot_costs_consumed)`, gerar `mile_entries` de saída apontando para os lotes consumidos (vários entries, ou uma entry agregada com array de consumos no `metadata`).
  - Transferência: debitar lotes do `from` account (consumo FIFO), criar lotes de entrada no `to` account com possível `bonusPercentage` (criar lotes `transfer_in` com `cost_basis_cents` proporcional ao custo transferido e com `bonusPoints` em lote separado), ajustar `program_accounts` saldos e custos.
  - Reversão: criar entry de reversão que recria lotes ou repõe `points_remaining` nos lotes originais com marcação de reversão, preservando auditoria.

- Transações e atomicidade:
  - Todas as operações mutadoras devem ocorrer em uma transação única: criar sale/transfer/purchase + criar/consumir lotes + criar entries + atualizar `program_accounts`.

- Consumo parcial de lote:
  - Ao consumir parte de um lote, atualizar `points_remaining` do lote, criar um registro de consumo (pode ser um `mile_entry` com `consumed_lot_id`).

6. Observações sobre o erro `$$id` e runtime Next.js

- Sintoma: ao importar Server Actions diretamente em rotas API (ou em módulos que são importados no server dev runtime) o dev server tenta re-declarar propriedades no proxy gerado para Server Actions, causando `TypeError: Cannot redefine property: $$id`.
- Mitigação imediata adotada: usar `await import()` dinâmico no handler de rota para evitar bundling ao iniciar; porém, logs de runtime ainda reportaram o erro quando a ação foi invocada indiretamente.
- Decisão técnica: separar a lógica transacional em `lib/services/movements.ts` que é um módulo normal (sem proxies de Server Actions). `app/*/actions.ts` (Server Actions) chamam esse service; `app/api/*/route.ts` e outros wrappers também chamam o mesmo service. Isso elimina import dinâmico/duplo e a dependência circular do runtime.

7. Riscos e pontos de atenção

- Migrações envolvendo novos índices/colunas podem exigir manutenção offline para grandes bases. Planejar janelas de manutenção.
- Reversões e correções de histórica serão mais simples se lotes preservarem origem e ids de transação.
- Testes de performance necessários: consumo de lotes em contas com muitos lotes (p.ex. centenas) deve ser otimizado via índice e queries limitadas.
